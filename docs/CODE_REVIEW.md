# Drift 代码审阅标准

本标准定义 drift 项目中"什么是 bug"、"如何分级"、"修复到什么程度算完成"，用于消除审阅判定歧义、避免 AI 因标准不统一陷入无限修复循环。

与 [CODE_STANDARDS.md](CODE_STANDARDS.md) 的分工：CODE_STANDARDS 规定"代码应该怎么写"，本文档规定"审阅时如何判定问题与终止修复"。

---

## 1. 核心原则

**契约优先**：bug 的判定以契约为唯一准绳。契约包括：
- 类型签名与函数签名
- 函数文档注释（godoc）
- 公开接口的方法语义
- [AGENTS.md](../AGENTS.md) 与 [CODE_STANDARDS.md](CODE_STANDARDS.md) 的明文规则
- [architecture.md](architecture.md) 中的设计契约

**未在契约中约定的，不属于 bug**。代码可以"更好"、"更优雅"、"更高效"，但只要不违反契约，就不是 bug，不进入修复流程。

---

## 2. bug 的严格定义

### 2.1 真 bug（必须修复）

| 类别 | 定义 | drift 项目典型例子 |
|------|------|-------------------|
| **正确性 bug** | 函数实际行为与文档/签名描述的契约不符 | `computeFileHashFromChunks` 不按 spec 拼接块哈希 |
| **崩溃 bug** | 在合法输入下 panic、nil deref、index out of range | `DetectEngine` 返回 nil 后调用方未判空 |
| **并发 bug** | race condition、deadlock、锁不可重入导致的死锁 | `SwitchBranch` 调 `AcquireWorkspaceLock` 二次加锁 |
| **资源 bug** | 文件句柄/goroutine/锁未释放 | `os.Open` 后未 `defer f.Close()` |
| **安全 bug** | 路径穿越、命令注入、未校验的用户输入 | 用户路径未过 `pathutil.RelToWorkDir` |
| **规范 bug** | 违反 AGENTS.md / CODE_STANDARDS.md 的**明文规则** | 文件超过 300 行、用 `%v` 包装 error |
| **数据完整性 bug** | 哈希校验缺失、原子写入缺失、半写状态 | `PutChunk` 未走 `WriteFileAtomic` |
| **架构 bug** | 违反 [architecture.md](architecture.md) 的设计契约：分层错误、循环依赖、跨层调用、抽象错位 | porcelain 直接操作存储细节而非通过 `Storer` 接口；filetype 反向依赖 porcelain |

### 2.2 非 bug（不进入修复流程）

以下情况**明确不算 bug**，审阅时记录为观察但不发起修复：

- **风格偏好**：变量命名"不够好"、注释"不够详细"、函数"可以更短"
- **性能优化机会**：未测量证实的"应该更快"、可读性优先的简单实现
- **过度防御**：为不可能的分支添加错误处理（违反 YAGNI）
- **抽象建议**：可以提取公共函数但没有重复三次以上
- **未实现功能**：roadmap 中的未来项（如 `LogsDir` 未启用），属于"未完成"而非"bug"
- **测试覆盖率不足**：除非有明文规则要求（如"导出函数必须有测试"），否则不算 bug
- **依赖版本升级**：除非有已知安全公告
- **TODO/FIXME 注释**：存在 TODO 不算 bug，除非该 TODO 指向已违约的契约

### 2.3 灰区判定规则

当无法立即判断时，按以下顺序裁决：

1. **有契约？** 查文档/签名 → 违反即 bug
2. **无契约但有测试？** 测试通过即非 bug
3. **无契约无测试？** 默认非 bug，除非属于 §2.1 崩溃/安全/数据完整性类别

## 3. bug 分级与响应

| 级别 | 定义 | 响应要求 | 例子 |
|------|------|---------|------|
| **P0 阻断** | 数据损坏、安全漏洞、生产环境崩溃 | 必须立即修复，优先于一切任务 | 哈希校验缺失、路径穿越、`PutChunk` 非原子写 |
| **P1 严重** | 功能错误、契约违反、并发死锁 | 当前会话内修复 | sentinel error 用字符串匹配、锁不可重入死锁 |
| **P2 一般** | 规范违反、防御性缺失、资源泄漏风险 | 批量修复，不阻塞主任务 | 文件超 300 行、未 `defer` 立即调用 |
| **P3 提示** | 风格、命名、注释完整性 | 仅记录不修复，除非用户要求 | 函数命名"不够精确"、注释"可以更清楚" |

**架构 bug 单独分级**：架构 bug 不按 P0-P3 分级，而是按 §4.4 架构修复流程处理。架构 bug 的优先级至少等同于 P1 —— 不得以"打补丁先绕过"为由拖延。

**审阅者不得擅自提升 P3 为 P2**。若想升级，需先在 [CODE_STANDARDS.md](CODE_STANDARDS.md) 添加明文规则。

## 4. 修复标准

### 4.1 修复必须满足的条件

1. **可验证**：修复后必须有测试覆盖，或现有测试已能验证修复效果
2. **最小化**：只改动与 bug 直接相关的代码，不顺手"清理周边"
3. **不破坏**：`go build ./...` + `go test ./...` + `go vet ./...` 全部通过
4. **不引入**：不引入新的 P0/P1 bug
5. **守规**：修复代码本身符合 CODE_STANDARDS.md

### 4.2 修复禁止的行为

- ❌ 修复 A bug 时顺手重构 B（"既然改了就一起改"）—— 普通 bug 修复严格守此条；架构 bug 修复按 §4.4 执行
- ❌ 为修复添加未被任何契约要求的新功能
- ❌ 为修复添加未被任何契约要求的新测试（除非该 bug 本身是测试缺失）
- ❌ 为修复添加未被任何契约要求的新依赖
- ❌ "防御性"添加本不需要的 nil 检查（违反 YAGNI）
- ❌ 把可工作的代码替换为"更优雅"的等价实现
- ❌ **对架构 bug 打补丁**：架构问题禁止用 workaround、特殊分支、临时绕过等方式修复，必须按 §4.4 修改架构

### 4.3 修复的边界

修复范围按 bug 类型区分：

**普通 bug 修复**（bug 局限于单个函数/模块）：
- bug 所在的函数/方法
- 该函数的直接调用方（仅当调用方依赖了 bug 行为时）
- 该函数的测试

**架构 bug 修复**（bug 根源在架构设计）：
- 按 §4.4 执行，改动范围以"消除架构违规"为边界
- 不扩大到无关重构

**超出上述范围的改动**必须单独发起，不在本次修复内完成。

### 4.4 架构修复原则

当 bug 的根因是架构设计问题（违反 [architecture.md](architecture.md) 的分层、依赖方向、抽象边界契约）时，必须通过修改架构修复，**严禁打补丁**。

**判定为架构 bug 的依据**：
- 违反分层方向（如 `storage` 反向引用 `porcelain`）
- 循环依赖（A → B → A）
- 跨层直接操作（如 `porcelain` 绕过 `Storer` 接口直接读写文件）
- 抽象错位（职责放错了层，如文件类型检测逻辑放在 `porcelain`）
- 同一职责在多个层重复实现且无共享祖先

**架构修复流程**：

1. **定位根因**：明确违反了 architecture.md 的哪条契约
2. **设计新结构**：提出符合契约的新结构，说明改动涉及哪些层/包
3. **最小化改动**：只移动与该契约相关的代码，不顺带优化无关部分
4. **保持接口**：对外公开接口签名尽量不变；若必须变，需同步更新所有调用方
5. **验证**：`go build ./...` + `go test ./...` + `go vet ./...` 全部通过；现有测试若因架构调整失败，需调整测试以符合新结构（不算"加无关测试"）

**架构修复禁止的行为**：
- ❌ 用 `if` 分支区分调用方来绕过分层错误
- ❌ 用接口断言（type assertion）跨层访问本不该可见的实现细节
- ❌ 用全局变量或 init 顺序掩盖循环依赖
- ❌ 用"先这样，以后再改"的方式拖延 —— 架构 bug 不存在"以后"

**架构修复与普通修复的边界**：若一个 bug 既有表面症状又有架构根因，先按架构修复消除根因，再评估是否还有普通 bug 需要单独修复。不要在同一轮中既改架构又改细节。

## 5. 终止条件（避免无限循环）

### 5.1 单次审阅的终止

满足以下**全部**条件，本次审阅终止：

- [ ] 所有 P0/P1 bug 已修复
- [ ] P2 bug 已修复或已记录为待办（每个 P2 必须有明确处理决定）
- [ ] P3 已记录但无需处理
- [ ] `go build ./...` exit 0
- [ ] `go test ./...` exit 0（已知的 Windows flaky 测试除外，见 AGENTS.md）
- [ ] `go vet ./...` exit 0
- [ ] 修复涉及的所有文件 ≤ 300 行（规范要求）
- [ ] 所有修复都符合 §4 修复标准

### 5.2 单个 bug 的修复终止

满足以下**任一**条件，该 bug 的修复终止：

- bug 已修复且测试通过
- bug 经调查**不成立**（属于 §2.2 非 bug），需在审阅记录中说明依据
- bug 经判定属于架构问题，已按 §4.4 架构修复流程处理完毕，或已在独立任务中启动架构修复且当前会话内无法完成（需在报告中明确记录剩余工作）
- bug 涉及未实现的契约（如 LogsDir 未启用），归类为"未来功能"而非 bug

### 5.3 明确禁止的循环模式

以下行为模式必须立即停止：

- **"修复后回归"循环**：修复 A 引发 B，修复 B 引发 A → 停止打补丁，说明存在架构 bug，转 §4.4 架构修复流程
- **"补丁叠加"循环**：为绕过架构问题不断添加分支/特例 → 停止，转 §4.4 架构修复
- **"标准升级"循环**：每修一个 bug 就发现新"bug" → 停止，说明 bug 判定标准过宽，需重新对齐 §2
- **"测试加码"循环**：修复一个 bug 后不断补测试 → 停止，测试只需覆盖该 bug 的契约，不加无关测试
- **"风格争议"循环**：就命名/写法反复讨论 → 停止，风格问题归 P3，不进入修复

## 6. AI 审阅行为规范

AI 在执行代码审阅时必须遵守：

### 6.1 必须做

- 审阅前先读 [AGENTS.md](../AGENTS.md) 与 [CODE_STANDARDS.md](CODE_STANDARDS.md)
- 每个 bug 标注级别（P0/P1/P2/P3）与依据契约
- 修复前先说明：bug 是什么、违反哪条契约、计划如何修
- 修复后跑 `go build` + `go test` + `go vet` 验证
- 修复完成后输出审阅报告（bug 列表、级别、处理结果）

### 6.2 禁止做

- 不主动修复 P3（除非用户明确要求）
- 不在同一轮审阅中修复超过 5 个独立 bug（避免上下文膨胀导致质量下降）
- 不修复"未实现功能"（归 roadmap）
- 不为**普通 bug**修复引入新的抽象层；架构 bug 修复按 §4.4 引入正确抽象
- 不修复"代码可以更好"（无契约依据的改进）
- 不对架构 bug 打补丁（见 §4.2、§4.4）

### 6.3 判定冲突时的处理

当 AI 与用户对"是否是 bug"有分歧时：

1. AI 引用契约依据（文档/签名/规则）
2. 若无契约依据 → AI 默认让步，归类为非 bug
3. 若有契约依据但用户坚持 → AI 执行用户决定，但在报告中记录分歧

## 7. 审阅报告格式

每次审阅结束时输出：

```
## 审阅报告

### 统计
- P0: X 个（已修复 X / 不成立 X / 待办 X）
- P1: X 个（...）
- P2: X 个（...）
- P3: X 个（仅记录）
- 架构: X 个（已修复 X / 进行中 X / 待办 X）

### 修复详情
1. [P1] <文件:行> <bug 简述>
   - 违反契约：<具体契约>
   - 修复方式：<改动摘要>
   - 验证：<测试名 / build/test/vet 结果>

2. [架构] <文件:行> <bug 简述>
   - 违反契约：<architecture.md 的具体条款>
   - 根因：<分层错误 / 循环依赖 / 抽象错位 / ...>
   - 架构修复方案：<新结构说明 + 涉及的层/包>
   - 验证：<build/test/vet 结果>

### 未修复（P2 待办）
- [P2] <文件:行> <bug 简述> → 原因：<需独立任务 / 不阻塞 / ...>

### 架构修复待办
- [架构] <bug 简述> → 剩余工作：<当前会话内已完成 X，剩余 Y 需独立任务>

### 验证结果
- go build ./... : exit 0
- go test ./...  : exit 0
- go vet ./...   : exit 0
```

## 8. 与其他文档的关系

| 文档 | 作用 | 与本文档关系 |
|------|------|-------------|
| [AGENTS.md](../AGENTS.md) | 项目速查规则 | 提供明文规则，违反即规范 bug |
| [CODE_STANDARDS.md](CODE_STANDARDS.md) | 编码规范详述 | 提供明文规则，违反即规范 bug |
| [architecture.md](architecture.md) | 架构契约 | 提供设计契约，违反即契约 bug |
| [roadmap.md](roadmap.md) | 路线图 | 区分"未实现功能"与"bug" |
| 本文档 | 审阅判定与终止 | 定义 bug、分级、修复、终止 |

当本文档与其他文档冲突时：**契约性文档（前四个）优先**，本文档仅规定审阅流程。
