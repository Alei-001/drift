# Drift - 产品需求文档

## 产品概述

**产品名称：** Drift

**产品定位：** 面向创意工作者的轻量级版本管理工具

**产品愿景：** 让创意工作者像程序员管理代码一样管理创意作品

## 目标用户

### 主要用户群体

| 用户类型 | 典型场景 | 核心痛点 | 文件类型 |
|----------|----------|----------|----------|
| **插画师/画画博主** | 多版本创作、客户选稿 | 手动管理文件混乱 | PSD、PNG、AI |
| **平面设计师** | 多方案探索、客户修改 | 版本多了找不到 | PSD、AI、SVG |
| **视频创作者** | 不同剪辑版本 | 大文件管理困难 | MP4、MOV |
| **摄影师** | 不同修图风格 | 同一原图多版本管理 | RAW、JPG |
| **小说作者** | 剧情分支、不同结局 | 版本命名混乱、无法对比 | TXT、Word、Markdown |
| **剧本创作** | 不同结局线、场景修改 | 难以对比修改内容 | TXT、Word、PDF |

> **重点**：小说作者和剧本创作者是核心受众之一。分支功能（平行剧情线）和文本 diff（查看改动）是刚需。

### 受众深度分析

| 维度 | 描述 |
|------|------|
| **主要文件类型** | 二进制大文件（.psd 几百MB、.blend 几GB、.ai、.tiff、视频素材）+ 文本（.txt、.md） |
| **工作方式** | 单人为主要，偶尔小型协作（不超过 2-3 人） |
| **技术背景** | 不理解 Git 概念（staging、commit、branch、merge） |
| **心智模型** | 「存一个版本 → 回到某个版本 → 导出给客户」 |
| **对工具的第一要求** | 简单、不碍事、不出错 |
| **不需要的能力** | merge（合并冲突是噩梦）、Git 的各种高级概念 |

### 设计原则

- 不要将 Git 的概念直接搬过来，用创意工作者理解的语言
- 二进制文件不能自动合并，所以不做 merge
- 大文件是常态，必须支持流式处理 + 进度反馈
- 默认行为要覆盖 90% 场景（减少手动 step）

### 用户痛点

现状：手动管理版本

```
文件夹：初稿/
文件夹：修改版1/
文件夹：修改版2/
文件夹：终稿/
文件夹：终稿_最终版/
文件夹：终稿_最终版_改/
文件夹：终稿_最终版_改_客户选这个/
```

**痛点：**
1. 命名混乱，找不到想要的版本
2. 占用大量磁盘空间（重复文件）
3. 无法对比两个版本的内容差异（小说作者）
4. 无法尝试不同方向并保留（设计师 / 作家）
5. 导出指定版本麻烦

## 功能取舍

### 必须保留/加强

| 功能 | 理由 | 实现状态 |
|------|------|----------|
| **save** | 核心功能 | ✅ 已实现 |
| **restore** | 「回到之前的版本」是刚需 | ✅ 已实现 |
| **export** | 导出给客户/合作方 | ✅ 已实现（dir/zip/tar.gz） |
| **status** | 需要让用户知道「哪些文件改了」 | ✅ 已实现 |
| **大文件支持** | 绝对核心。进度条、流式读写、不 OOM | ✅ 已实现 |
| **ignore** | 跳过临时文件、渲染缓存、自动保存文件 | ✅ 已实现（.driftignore） |
| **config** | 需要用户名，用于记录谁保存的版本 | ✅ 已实现 |
| **diff** | 对文本（作家/文案）有用 | ✅ 已实现（文本 diff，二进制显示大小变化） |
| **branch** | 「试不同的配色方案/剧情线」 | ✅ 已实现（平行版本线，无 merge） |
| **switch** | 切换分支 | ✅ 已实现（支持 `--create` 自动创建） |
| **unstage** | 取消暂存 | ✅ 已实现 |
| **log** | 查看提交历史详情 | ✅ 已实现 |

### 已实现命令清单

| 命令 | 功能 | 描述 |
|------|------|------|
| `drift init` | 初始化 | 在当前目录创建 `.drift/` 项目结构 |
| `drift add` | 暂存 | 添加文件/目录到暂存区 |
| `drift status` | 状态 | 显示工作区、暂存区、提交状态 |
| `drift unstage` | 取消暂存 | 清空暂存区内容 |
| `drift save` | 保存 | 将暂存区保存为新版本 |
| `drift list` | 列表 | 查看版本历史（含分支信息） |
| `drift log` | 日志 | 查看详细提交历史 |
| `drift export` | 导出 | 导出版本为目录/zip/tar.gz |
| `drift restore` | 恢复 | 将工作区恢复到指定版本 |
| `drift branch` | 分支 | 创建、删除、重命名、列出分支 |
| `drift switch` | 切换 | 切换到指定分支 |
| `drift diff` | 对比 | 显示版本/文件差异 |
| `drift config` | 配置 | 设置或查看配置选项 |

### 可保留但需简化

| 功能 | 理由 | 策略 |
|------|------|------|
| **add / staging** | 创意工作者大概率要么全存要么不存。staging 的价值是「保存前预览」 | 保留，未来可考虑默认 add all |

### 明确不需要（过度工程）

| go-git 功能 | 为什么不需要 |
|-------------|-------------|
| packfile / delta 压缩 | 二进制文件不压缩，存储便宜 |
| merge / conflict resolution | 单人/极小团队，不做 merge |
| transport / remote protocol | MVP 阶段纯本地 |
| reflog | 过于 Git 内部概念 |
| rebase / cherry-pick / stash | 开发者工具 |
| tag | version ID 已足够，不需要两层命名 |
| GPG signing | 创意工作者不关心 |
| submodule | 不需要 |
| shallow clone | 不需要 |
| 复杂的 ref-parse 规则 | 不需要 `refs/heads/`, `refs/remotes/` 等 |

### 未来可能新增（go-git 没有的）

| 功能 | 理由 |
|------|------|
| **自动保存** | 「我改了就自动存一个版本」，接近 Dropbox/Figma 的体验 |
| **缩略图/预览** | 版本列表带缩略图，创意工作者按视觉识别版本 |
| **项目模板** | `drift init --template illustration` 预设 ignore 规则 |

## 用户工作流

### 基本工作流

```bash
# 初始化项目
drift init

# 修改文件后查看状态
drift status

# 添加文件
drift add .

# 保存版本（message 可选）
drift save
drift save -m "完成前四章"

# 查看历史
drift list

# 对比版本
drift diff v1 v2

# 导出版本
drift export v1 -o ./交付客户

# 回退到旧版本
drift restore v2
```

### 分支工作流（作家示例）

```bash
# 在当前版本基础上创建剧情分支
drift branch 悲剧结局
drift switch 悲剧结局

# 在分支上创作
drift add .
drift save -m "悲剧结局初稿"

# 切回主线继续
drift switch main

# 创建另一个结局
drift branch 美满结局
drift switch 美满结局
drift add .
drift save -m "美满结局初稿"

# 对比两个结局
drift diff 悲剧结局 美满结局

# 查看所有分支
drift branch list
```

## 使用场景

### 场景 1：画画博主

1. 画师开始画稿，保存版本 v1（初稿）
2. 继续修改，保存版本 v2（修改版）
3. 客户要求尝试不同风格
4. 创建分支"风格A"，在分支上工作
5. 客户选择 v1，导出 v1 交付

### 场景 2：小说作者

1. 作者写完前三章，保存版本 v1
2. 继续写第四、五章，保存版本 v2
3. 想尝试不同剧情走向
4. 分别创建分支"结局A"和"结局B"
5. 在两个分支上独立创作
6. 用 `drift diff` 对比两个版本
7. 最终选择一个结局，导出完整版本

### 场景 3：设计师

1. 设计师完成初稿，保存版本 v1
2. 客户要求修改配色
3. 保存版本 v2（修改配色）
4. 客户又想看另一个方案
5. 从 v1 创建分支"方案B"
6. 最终客户选择 v2，导出交付

## 成功指标

| 指标 | 目标 |
|------|------|
| **学习成本** | 10 分钟内上手基本操作 |
| **操作效率** | save / restore 覆盖日常高频操作 |
| **用户满意度** | 比手动管理文件夹更方便 |
